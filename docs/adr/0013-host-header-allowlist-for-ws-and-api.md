# Host-header allowlist for /ws and /api: a second, independent layer beyond the Origin check

`/ws`'s WebSocket upgrader no longer overrides `CheckOrigin`; it uses gorilla's default (allow when no `Origin` header is present, otherwise require `Origin` to equal the request `Host`). On top of that, `/ws` and every `/api/*` route are wrapped in a Host-header allowlist (`internal/server/hostallowlist.go`, `AllowedHosts`/`hostAllowlist`) that rejects a request carrying an `Origin` header whose `Host` isn't on the allowlist. Implemented in #164 (issue #163).

**Why two layers.** They defend against two different attacks, and neither alone is sufficient.

- **Plain cross-origin read.** A page on `evil.com` opens `ws://localhost:8080/ws` (or fetches `/api/...`) from the victim's own browser. Before this fix, `/ws`'s `CheckOrigin` unconditionally returned `true`, so any origin could open the WebSocket and read the full live cluster topology — node names, namespaces, pod names, phases. This works on htp-k8s's **loopback default** ([ADR-0012](0012-secure-by-default-network-binding.md)) — the attacker needs no network reach to the port at all; the victim's browser supplies it. Deleting the override and letting gorilla's own same-origin default apply closes this for `/ws`: `Origin` (`http://evil.com`) doesn't equal `Host` (`localhost:8080`), so the upgrade is rejected. `/api` was never vulnerable to this shape in the first place — no `Access-Control-Allow-Origin` header is set anywhere, so the browser's own CORS/SOP already blocks `evil.com`'s JS from reading the response body (the request still executes server-side, same as any cross-site GET, but nothing sensitive leaks back to the attacker's page).

- **DNS rebinding — the part a same-origin check cannot fix.** An attacker points a DNS name they control at a short TTL, gets the victim's browser to load a page from it, then rebinds the name to the target address (`127.0.0.1`, or wherever htp-k8s is actually reachable from). The browser's *next* request to that same page's origin — now resolving elsewhere — carries `Origin: http://evil.com:8080` **and** `Host: evil.com:8080`. Those match each other, so `checkSameOrigin` (or any same-origin check) allows it: same-origin is a property of the request's own headers, and rebinding makes both of them equally attacker-controlled. This is also why `/api`'s "no CORS header" protection doesn't hold under rebinding: the browser now considers `evil.com:8080` the same origin as the response, and its own JS *can* read it. Closing this needs to ask a different question — not "do `Origin` and `Host` agree with each other" but "is `Host` one we actually recognise" — which is what the allowlist checks, independent of whatever the request itself claims.

**Why this bites the loopback default specifically, and isn't covered by [ADR-0012](0012-secure-by-default-network-binding.md).** ADR-0012 reasons about who can reach the *port* — its whole mitigation is "nothing off this machine can open a connection at all." DNS rebinding doesn't need off-machine reach: the attacker's page runs in the victim's own browser, on the victim's own machine, and it's the *victim's browser* that ends up dialing `127.0.0.1` after the rebind. ADR-0012's reasoning explicitly doesn't cover this axis — hence a separate ADR rather than folding this into it. Cross-reference: ADR-0012 links back here in its own text.

**Only /ws and /api are wrapped**, not `/healthz` or the static frontend at `/` — neither carries cluster data, and `/`'s own load is a top-level navigation, which browsers don't send an `Origin` header for anyway.

**Requests with no `Origin` header are never gated** by the allowlist — on both `/ws` and `/api`, matching gorilla's own no-`Origin` allowance. This is a deliberate, not incidental, choice: non-browser clients (`curl`, other tooling, monitoring, the e2e harness) are not this ticket's threat model, and an attacker's page can't make the victim's browser *omit* the `Origin` header a WebSocket upgrade or a `fetch`/`EventSource` call would otherwise carry — so there is no rebinding-shaped attack hiding behind this exemption.

## What's trusted without configuration

`server.NewAllowedHosts(addr, extraHosts)` builds the allowlist:

- **Loopback host forms** (`localhost`, `127.0.0.1`, `::1`/`[::1]`, any port) are **always** trusted. An attacker's DNS can make a browser *resolve* an attacker-chosen name to the target's address, but it can never make the browser send a `Host` header whose hostname *text* is literally one of these — that text comes from the URL in the victim's address bar, which the attacker doesn't control.
- **The host `-addr` binds to, when it names a concrete IP literal** (e.g. `-addr 192.168.1.5:8080`) is trusted automatically, for the same reason: a request whose `Host` genuinely equals the server's own real bind address was, by construction, addressed straight at the server, not routed through a rebound name.
- **A wildcard bind** (`-addr :8080`, no host) derives nothing beyond the loopback forms above — it cannot know its own reachable name. This is the case `-allowed-hosts`/`HTP_K8S_ALLOWED_HOSTS` exists for, alongside reverse-proxy and DNS-name deployments where the public-facing hostname is never the bind address at all. A rejection in this shape logs the offending `Origin`/`Host` and names `-allowed-hosts` explicitly in both the log line and the HTTP error body, rather than presenting as an opaque 403 — a reverse-proxy operator hitting this needs to be able to diagnose it from the log alone.

**Why this must not reinstate a configuration step.** #160 removed the need to rebuild the frontend with `VITE_WS_URL`/`VITE_API_URL` for local development and same-machine loopback use — the frontend now derives its target from `window.location`. This fix deliberately preserves that: the loopback and addr-derived trust above cover every case #160 unblocked with zero added configuration. `-allowed-hosts` is additive, only reached by deployments that were never covered by the loopback default in the first place (reverse proxies, DNS names, wildcard binds for remote viewing).

## Rejected alternative: fold this into ADR-0012

Origin/Host trust and bind-address exposure are different axes. ADR-0012 governs *who can open a TCP connection to the port at all*; this ADR governs *which already-connected browser requests get treated as trustworthy*, which — per the rebinding case above — is a question that has to be answered even on the loopback default ADR-0012 already secures. Keeping them separate keeps ADR-0012 about the bind address and this one about request trust, rather than overloading one ADR with two independent decisions.

## Revisit if

htp-k8s ever grows a reverse-proxy-aware deployment mode with a canonical public hostname baked in some other way (e.g. read from a config file rather than a flag) — at that point `-allowed-hosts` might become redundant with that source rather than the sole configuration surface for it.
