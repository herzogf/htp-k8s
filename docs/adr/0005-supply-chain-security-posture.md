# Supply-chain security: SBOM, CVE scanning, and attestation posture

Given the project's supply-chain sensitivity (see [[0001]]) and the goal of giving users confidence in released artifacts, we adopted: GoReleaser's built-in `sboms` block (Syft, CycloneDX) to generate an SBOM for every release artifact; `govulncheck` (Go) and `npm audit` (frontend) for source-level CVE scanning on every PR; Trivy scanning the `ko`-built container image (built locally via `ko build --local`, no registry push) on every PR *and* at release, with SARIF uploaded to GitHub's Security tab so new findings are annotated directly on the PR; and GitHub's native artifact attestations (`actions/attest-sbom`, `actions/attest-build-provenance`) for keyless, Sigstore-backed signing of the SBOM and build provenance, verifiable via `gh attestation verify` with no key management required.

We distinguish a CVE **finding** from a scan **operation failure**. CVE findings are **report-only, never blocking** — a Critical/fixable finding surfaces immediately (PR annotation / release notes, via `exit-code: "0"`) but does not fail the job. Blocking on findings was rejected because it risks the project being unable to ship over an unfixable upstream CVE in a base image or transitive dependency; report-only still gives upfront, PR-level visibility without that operational risk.

A scan that **fails to run**, however, **does** fail the job: if Trivy cannot execute (registry auth/pull error, DB-download failure, hang → step `timeout-minutes`) or the SARIF upload fails, the job errors rather than silently producing an empty Security tab. We always want a real scan result, so a broken scan is not tolerated. Concretely: the Trivy steps carry no `continue-on-error` (only `exit-code: "0"` keeps *findings* non-fatal), they have step timeouts, and the PR-time **`Container scan (Trivy)` job is a required status check** — a scan that can't run blocks the PR, catching it before it would fail a release (the release workflow fails on the same). Findings never block; a broken scan does.

We chose GitHub's native attestation actions over configuring GoReleaser's own cosign-signing block, to avoid running two overlapping signing mechanisms for no added benefit — GitHub's OIDC-based keyless signing needs no key management and is directly verifiable by end users via the `gh` CLI.

**Multi-arch scan and attestation scope (issue #95).** The released container image is a multi-arch index (`ko build --platform=linux/amd64,linux/arm64`); scanning or attesting only the index resolves to a single, runner-default architecture, silently leaving the other arch unscanned/unattested. We draw the scope line differently at the two stages, deliberately: the **PR-time** pre-check (`build.yml`'s `Container scan (Trivy)` job) builds and scans a single-arch (amd64) local image via `ko build --local` — it exists to give fast, PR-level feedback, not to be the authoritative multi-arch gate, so single-arch is an acceptable, cheaper trade there. The **release-time** scan and attestation, by contrast, are the authoritative safety net and cover **both** architectures: `release.yml`'s `image` job parses the per-platform child digests out of ko's `--image-refs` output (index on line 1, children after), and fans them out — via matrix jobs mirroring the existing per-archive SBOM attestation fan-out — to (a) an independent Trivy scan per child digest, each SARIF uploaded to the Security tab under its own category, and (b) an independent `actions/attest-build-provenance` call per child digest, alongside the pre-existing index-level attestation. Nothing here hardcodes the platform list; both fan-outs are driven by however many child digests ko actually produced, so they track the `--platform` flag automatically. The index attestation remains the one `gh attestation verify oci://<image>:<tag>` uses (a tag resolves to the index); the per-child attestations make verifying one specific platform's digest possible too, which the index-only attestation could not do.

**Moving image tags (issue #64).** A stable release (`vX.Y.Z`, no `-` in the tag) publishes the exact tag plus `vX.Y`, **if this release is the newest patch within that minor** (true for a normal release; deliberately NOT true, and so NOT published, when re-tagging an older patch after a newer one in the same minor already shipped — publishing vX.Y then would move it *backwards*), and, once `X >= 1`, bare-major `vX` — and `latest`/`vX` publish only when this release is additionally the **highest stable version that exists overall**, so a backport patch to an older line can never repoint them backwards (the previous logic moved `latest` for *any* stable tag, unconditionally — untriggered so far only because every release has been linear). Every moving tag keeps the project's own `v` prefix (decided after #64's issue text used bare `1.2` in a passing example and nobody flagged it as a decision during grilling): the exact tag, the git tags, the GitHub releases, the changelog, and every README quickstart are all `v`-prefixed already, so a bare `X`/`X.Y` would be the one inconsistent form a consumer sees — a user who copies `:v0.3.1` and wants to loosen to the minor line would type `:v0.3`, which fails outright under a bare scheme. `latest` alone stays bare: it isn't a version, so there's no `v` to add, and it doesn't need one to signal "this moves" the way the bare-forms argument assumed — `latest` already carries that signal on its own. Bare-major `0`/`v0` is deliberately never published regardless of prefix: SemVer gives no compatibility guarantee across `0.x` MINORs, so a moving major-0 tag would imply a stability guarantee that doesn't exist, whereas `vX.Y` stays honest even pre-1.0 since PATCH releases remain backward-compatible there. `X >= 1` means `v1.0.0` starts emitting `v1` with no code change — the rule is symmetric around the 1.0 boundary by construction, not special-cased. A prerelease (tag contains `-`) publishes its exact tag only — no moving tag, no `latest` — since it is a pinned, one-off artifact by design. This logic lives in `.github/scripts/compute-image-tags.mjs`, a pure function with its own unit-test suite that runs in PR CI, specifically because `release.yml` is tag-triggered and can never be exercised by a PR itself (the issue #156 lesson). Moving tags are a **pull convenience, not a verification target**: `gh attestation verify` resolves a tag to a digest at verification time, so verifying `:latest`/`:vX`/`:vX.Y` only proves *some* release matching that tag is authentic, not a specific one — the exact tag (or a digest resolved from a moving tag beforehand) is what verification examples use (README "Supply chain").

**Release-integrity model (issue #95).** The whole release process trusts that the tagged commit already passed CI — `release.yml` only builds and publishes, it does not re-run tests. Two changes make that trust assertable rather than implicit: (1) the CI workflow (`build.yml`, renamed from `pr.yml`) now runs on `push: branches: [main]` in addition to `pull_request`, so every commit that lands on `main` gets its own check-runs (previously only PR head commits did, never the post-merge `main` commit a tag would point at); (2) `release.yml` gained a first job, gating every other job via `needs:`, that reads the tagged commit's check-runs (`gh api .../commits/<sha>/check-runs`) and requires each required check to have concluded `success` or `skipped` — a job-level skip (e.g. the heavy jobs on a docs-only `main` push) counts as passing, mirroring how branch protection itself treats a skipped required check (issue #76). It polls with a bounded timeout rather than reading once, since a tag can be pushed moments after the merge, before `main`'s CI has finished; a real failure is reported immediately without waiting out the timeout. This turns "tag an untested or off-`main` commit" from a silent risk into a fast, loud release failure.

Dependabot (`.github/dependabot.yml`) is configured with a **cooldown** period for npm (5 days default/patch, 7 days minor, 14 days major) — deliberately *not* fast-tracking patch-level bumps, since several real npm supply-chain attacks shipped their payload as a patch-version bump specifically to exploit low-scrutiny auto-update paths. Cooldown only delays routine version updates; Dependabot's security-alert-triggered updates (CVE fixes) are unaffected and still land immediately. Go modules are left without a cooldown for now, matching Go's materially different supply-chain risk profile (smaller/more static dependency trees, checksum-verified via GOSUMDB by default).
