# Supply-chain security: SBOM, CVE scanning, and attestation posture

Given the project's supply-chain sensitivity (see [[0001]]) and the goal of giving users confidence in released artifacts, we adopted: GoReleaser's built-in `sboms` block (Syft, CycloneDX) to generate an SBOM for every release artifact; `govulncheck` (Go) and `npm audit` (frontend) for source-level CVE scanning on every PR; Trivy scanning the `ko`-built container image (built locally via `ko build --local`, no registry push) on every PR *and* at release, with SARIF uploaded to GitHub's Security tab so new findings are annotated directly on the PR; and GitHub's native artifact attestations (`actions/attest-sbom`, `actions/attest-build-provenance`) for keyless, Sigstore-backed signing of the SBOM and build provenance, verifiable via `gh attestation verify` with no key management required.

We deliberately made CVE scanning **report-only, never blocking** — a Critical/fixable finding surfaces immediately (PR annotation or release notes) but does not fail the job. Blocking was rejected because it risks the project being unable to ship any release over an unfixable upstream CVE in a base image or transitive dependency; report-only still gives upfront visibility (via PR-level scanning, not just release-level) without that operational risk. This is a one-line change to tighten later (set the Trivy step's exit-code threshold) if CVE hygiene proves solid enough to justify hard gating.

We chose GitHub's native attestation actions over configuring GoReleaser's own cosign-signing block, to avoid running two overlapping signing mechanisms for no added benefit — GitHub's OIDC-based keyless signing needs no key management and is directly verifiable by end users via the `gh` CLI.

Dependabot (`.github/dependabot.yml`) is configured with a **cooldown** period for npm (5 days default/patch, 7 days minor, 14 days major) — deliberately *not* fast-tracking patch-level bumps, since several real npm supply-chain attacks shipped their payload as a patch-version bump specifically to exploit low-scrutiny auto-update paths. Cooldown only delays routine version updates; Dependabot's security-alert-triggered updates (CVE fixes) are unaffected and still land immediately. Go modules are left without a cooldown for now, matching Go's materially different supply-chain risk profile (smaller/more static dependency trees, checksum-verified via GOSUMDB by default).
