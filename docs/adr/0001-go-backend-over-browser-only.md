# Use a Go backend instead of browser-only + kubectl proxy

The simplest possible setup is a static frontend calling `kubectl proxy`'s local HTTP port directly, with no server component. We rejected this because two goals need a real server: (1) real exec-triggered pod activity requires receiving a Kubernetes audit log webhook, which only a running server process can do — a browser cannot receive a webhook; and (2) the hosted version planned for later cannot safely hand cluster credentials to a stranger's browser, so it needs a backend regardless. Building the backend now means the local and hosted versions share one architecture instead of the local version being rewritten later.

We chose **Go** over Node.js for this backend, despite the frontend (Three.js/React) necessarily being a Node/npm-built bundle. The backend is the part holding live cluster credentials in a long-running process — the highest-value target for a supply-chain attack — and recent npm ecosystem incidents made that risk unacceptable there. Go offers `client-go`, the reference Kubernetes client library (with mature OpenShift extensions), compiles to a single static binary with the built frontend assets embeddable via `go:embed`, and its toolchain doesn't execute arbitrary install-time scripts the way npm's `postinstall` does. The cost: the user knows Java, not Go, and will rely on AI assistance for backend changes rather than writing them unaided — accepted as worth the security and distribution benefits. Java/Spring Boot was the runner-up (developer's own expertise, decent `fabric8 kubernetes-client`) but loses on packaging simplicity (JRE or native-image friction) and startup time.
