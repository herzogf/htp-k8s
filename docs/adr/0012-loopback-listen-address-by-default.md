# The server listens on loopback only by default

`htp-k8s` has no authentication or authorization layer anywhere in `internal/server/` — [[0003-cinematic-viewer-not-admin-tool]] keeps it a read-only cinematic viewer, not a reason to add one. Prior to this decision the default listen address was a bare `:8080` (all interfaces): anyone who could reach that port got a full read-only view of the operator's cluster, under the operator's own credentials. htp-k8s's default use case is **local** — the browser runs on the same machine as the server, whatever the *cluster* itself is (this holds even for the conference-showcase Demo Mode) — so a wide-open default bought nothing for that case and cost real exposure for everyone else.

We changed the default to `127.0.0.1:8080` (loopback only). `-addr`/`HTP_K8S_ADDR` remain the explicit, deliberate opt-in to expose (e.g. `-addr :8080`). The startup log always states the listen address and, when it's loopback, names both opt-out spellings — an operator finds out from the log, not by discovering later that the app was unreachable, or reachable, from somewhere they didn't expect.

## Breaking change, but narrower than it looks

This does **not** newly break the primary "load the page in a browser on another machine" case — that was already broken on the old all-interfaces default too, because the released frontend's WebSocket target is baked to `ws://localhost:8080/ws` regardless of where the page was loaded from (`DEFAULT_WS_URL` in `web/src/config.ts`), so a remote browser already got a blank, stalled page there, not a working live scene. What silently breaks on upgrade: anyone reaching `/api` endpoints directly from another host (curl, custom tooling, monitoring), and anyone who had already rebuilt the frontend with a matching `VITE_WS_URL` to make real remote viewing work — both now also need `-addr`/`HTP_K8S_ADDR`.

## The container image keeps a wider effective default — deliberately, as a recipe flag, not baked into the binary

The published container image's **binary** has the same loopback default as everywhere else — it is not an exception. What *is* an exception is the documented `docker run` recipe (README.md), which passes `-e HTP_K8S_ADDR=:8080` explicitly. Docker's `-p` port publishing forwards to the *container's own* interface address, never to its loopback, so a container left on the loopback default would never see traffic `-p` forwards to it — breaking `docker run -p 127.0.0.1:8080:8080 ...` even though that command's host-side loopback binding is what actually restricts access. The container just has to bind wider *inside its own network namespace* for that forwarding to reach it.

Two alternatives were considered and rejected:

- **A container-*runtime* environment variable baked into the image itself**, so the recipe would need no extra flag at all. Not possible with the current toolchain: verified against ko v0.19.1 and GoReleaser v2.17.0 source (this repo doesn't even use GoReleaser's `kos:` pipe for the image — it builds with plain `ko build`, configured by `.ko.yaml`, entirely separate from GoReleaser). Both `.ko.yaml`'s own `builds[].env` and GoReleaser's `kos[].env` feed straight into `go build`'s environment — Go *build*-time env, not the OCI image's runtime `Config.Env`. The only two entries `ko` ever writes into an image's real runtime `Config.Env` are two hardcoded internal values (`KO_DATA_PATH`, `PATH`); there is no exposed flag or config field for anything else.
- **An image-only *build*-time default** (baking `-X main.defaultAddr=:8080` into just the container build via `.ko.yaml`'s own `ldflags:` block — mechanically possible, `.ko.yaml` already has one, `defaultAddr` would only need to become a `var` instead of a `const`). This one **is** achievable, and was rejected anyway: it would make the `--network host` local-cluster fallback recipe (below) fail **open** instead of closed. Every other way to forget a flag in this design loses connectivity, not access control — an operator who forgets `-addr`/`HTP_K8S_ADDR` on the native binary, or forgets `-e HTP_K8S_ADDR=:8080` on the bridge-networked container recipe, just can't reach the app. But an operator who forgot `-e HTP_K8S_ADDR=127.0.0.1:8080` on the `--network host` fallback, if the image's *own* default had quietly widened to `:8080`, would silently expose every interface on their host — the one shape this design goes out of its way not to have. Keeping the env var explicit means that forgetting it there fails exactly the same way as everywhere else: loudly, by not working, not silently, by being exposed.

## Local clusters need one more step, and it isn't `--network host`

A local cluster's kubeconfig usually points at `https://127.0.0.1:<port>` — meaningless from inside a bridge-networked container, which has its own loopback distinct from the host's. The obvious fix, `--network host`, is the wrong default answer: it shares the host's network stack outright, so Docker's `-p` publish mapping is silently ignored — a container bound to a port under `--network host` is bound to that port on the *host*, across every interface, regardless of what `-p` says. That is exactly the exposure this whole decision exists to close, so it cannot be the universal local-cluster recipe, even though it is the only *general* one available (some local tools — k3d, minikube with the Docker driver, Docker Desktop's built-in Kubernetes — have no better option, so the fallback recipe keeps it, with the loopback bind moved into the app itself via `-e HTP_K8S_ADDR=127.0.0.1:8080`).

kind is the one case with a real alternative: its own Docker network (`--network kind`) plus its *internal* kubeconfig (`kind get kubeconfig --internal`) let ordinary bridge networking plus `-p 127.0.0.1:<port>:8080` reach the cluster — verified end-to-end against a real kind cluster and a real `ko`-built image, with the host listener staying genuinely loopback-only (`ss -ltn`, and a failed connection attempt from the host's real LAN IP).

See [Running & developing locally](../running-locally.md#the-container-image) for the actual commands and the practical detail behind both approaches; this ADR is the reasoning, not the recipe.
