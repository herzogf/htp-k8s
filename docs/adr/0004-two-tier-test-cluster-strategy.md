# Two-tier fake-cluster strategy for testing: kind + KWOK

Testing this app meaningfully requires real pod lifecycle behavior (phase transitions, restarts, Events) that neither a fake clientset nor `envtest` can produce — both skip the kubelet entirely, so pods never actually run. A real cluster is needed for that. But the app also must handle large-scale clusters (~50 nodes, thousands of pods), and running that many real kind node-containers is infeasible on both developer machines and GitHub-hosted CI runners (each real kind node costs real, per-node memory).

We adopted two tiers instead of one: a real, single-node **kind** cluster (created programmatically via kind's Go library from test code, no manually-installed `kind` CLI or pre-existing cluster needed — only Docker) for validating actual Kubernetes behavior against a handful of real pods; and **KWOK** (Kubernetes WithOut Kubelet, CNCF sandbox project), run as a controller *attached to that same kind cluster* (not a separate fake-cluster stack), adding simulated Node/Pod objects on top of the real ones. Rejected alternatives: real multi-node kind for scale testing (too memory-heavy), `envtest` (no kubelet, same limitation as fake clientset for our purposes), fake-clientset-only testing (never validates real watch/event behavior end-to-end), `kwokctl`'s fully-separate fake control plane (unnecessary second cluster stack when KWOK-as-controller can augment the kind cluster already running).

KWOK is used at two different scales for two different purposes: a **modest** amount (5–10 extra nodes, 30–50 extra pods) is added in the same PR-blocking e2e job, purely to make Playwright's screenshots/video show a visually populated scene cheaply and reliably. The **full target scale** (50+ nodes, thousands of pods) is reserved for a separate nightly/scheduled job — not because it's expensive, but because rendering that many instanced panels via headless Chromium's software WebGL rendering is inherently more timing-variable, and a scale/performance check shouldn't risk blocking an unrelated PR on CI flakiness.

**Status update (issue #29/#171/#174, 2026-07):** the nightly job (`.github/workflows/nightly.yml`, `test/e2e/kwok/seed-scale.sh`) initially shipped (#171) with a smaller, rehearsed scheduled default (15 fake nodes, ~1,231 pods) after an earlier, never-rehearsed 50-node default surfaced real problems the first time it was actually run end to end. **#174 has since closed that gap:** 50 fake nodes / 3,671 seeded pods is now the rehearsed, shipped scheduled default, meeting this ADR's "50+ nodes, thousands of pods" target — see `nightly.yml`'s and `seed-scale.sh`'s own header comments for the rehearsal evidence and the measured numbers, which are that pair's home, not this ADR's. The one residual constraint is `hot_pod_count` (the deliberately dense Tower's pod count), still held at 260: SCENE height keys off the busiest Tower, so raising node count alone was safe, but raising hot pod count is what triggers a pre-existing `demoMode.ts` climb-rate limitation, tracked separately as #173 and still open. This paragraph records that history rather than silently rewriting the decision above.
