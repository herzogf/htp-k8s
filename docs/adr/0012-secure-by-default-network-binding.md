# Secure-by-default network binding: loopback by default, exposure is an explicit opt-in

htp-k8s listens on `127.0.0.1:8080` by default; `-addr`/`HTP_K8S_ADDR` is the explicit opt-in to expose it wider (e.g. `-addr :8080`). Implemented in #132 (issue #127).

**Why.** There is no authentication or authorization layer anywhere in `internal/server/`. Per [[0003-cinematic-viewer-not-admin-tool]] this is a read-only viewer, not an admin tool — but read-only is not harmless when nothing is checking who's asking: a non-loopback bind hands anyone who can reach the port a live view of the operator's cluster under the operator's own credentials. And the default use case is local: the browser runs on the same machine as the server regardless of whether the *cluster* itself is local or remote — this holds even for the conference-showcase Demo Mode, which also runs on a laptop. Binding wider by default buys nothing for that case and costs real exposure for everyone else.

**The container asymmetry.** One default cannot serve both of Docker's networking modes:

- **bridge + `-p 127.0.0.1:8080:8080`** (the documented default for a remote cluster, or a local kind cluster reached via `--network kind` + `kind get kubeconfig --internal`) — the app inside the container must bind `0.0.0.0`/`:8080`, or Docker's published port never reaches it. Still safe: the host-side `-p` publish, not the process's own bind, is what restricts access.
- **`--network host`** (the fallback for a local cluster with no kind-style in-network hostname, e.g. k3d, minikube, Docker Desktop) — there is no separate network namespace, `-p` is silently ignored, and a `:8080` bind *inside* the container is a `:8080` bind on the host, across every interface. `HTP_K8S_ADDR=127.0.0.1:8080` is mandatory here, not optional.

The actual `docker run` recipes for each case live in README.md and `docs/running-locally.md`, not duplicated here.

**Organizing principle: every forgetting-path fails closed.** Forget `-e HTP_K8S_ADDR=:8080` under bridge networking → the published port has nothing listening behind it, connection reset. Forget `HTP_K8S_ADDR=127.0.0.1:8080` under `--network host` → the binary's own default is already loopback, so the bind stays loopback. Both failure modes are merely annoying, never a silent exposure. This property is why the image-default alternative below was rejected.

**Rejected alternatives.**

- **Detect "am I in a container?" and bind `:8080` there.** Tried once already and reverted: #124 removed a uid-65532 heuristic in review because it silently false-negatived under `docker run --user`, and #131 has since made `--user "$(id -u):$(id -g)"` part of the *documented* recipe — so any uid-based sniffing would now be actively wrong on the standard invocation, not just a fragile edge case. Configuration, not sniffing.
- **Bake a wider default into the image itself.** An image-only build-time default (e.g. `-ldflags -X main.defaultAddr=:8080` in `.ko.yaml`, which already has its own `ldflags:` block, distinct from GoReleaser's) is mechanically possible, and would remove the `-e HTP_K8S_ADDR=:8080` flag from the documented bridge-mode recipe. Rejected because it inverts the failure mode above for the `--network host` path: a container whose own baked-in default is already `:8080` would expose every interface on the host the moment someone omitted `HTP_K8S_ADDR=127.0.0.1:8080`, silently, rather than failing closed. Recorded explicitly as *possible but rejected*, not impossible — see Consequences for the separate, genuinely-impossible option this isn't.

**Consequences.** The bridge-mode container recipe carries one extra flag, `-e HTP_K8S_ADDR=:8080` — not purely a chosen cost: there is no config-level way to set a container-*runtime* env var through either plain `ko`'s `.ko.yaml` or GoReleaser's `kos:` pipe (both only expose Go **build**-time env — verified against `ko` v0.19.1 and GoReleaser v2.17.0; the `kos:` pipe was checked for the mechanism only, this repo does not use it), so a flag-free recipe by image configuration was never on the table; the only route to one was the ldflags default rejected above. Remote *browser* viewing needs more than a wider `-addr`/`HTP_K8S_ADDR` regardless: the released frontend's WebSocket target is baked in at build time (`ws://localhost:8080/ws`, `DEFAULT_WS_URL` in `web/src/config.ts`), so a stock build never reaches a remotely-running backend — real remote viewing has always required a frontend rebuilt with `VITE_WS_URL`, independent of the server's bind address. So this change's actually-affected population is narrower than "everyone who viewed remotely": direct `/api` callers (curl, custom tooling, monitoring) that relied on the old all-interfaces default, and the presumably small set of setups that had already done that frontend rebuild to make remote viewing work.

**Revisit if** htp-k8s ever grows an authentication layer — at that point a non-loopback default stops meaning "read access under your credentials with nothing checking who's asking," and the calculus changes. Revisit the flag-free recipe ruled out above if `ko` or GoReleaser gain a container-runtime env surface.
