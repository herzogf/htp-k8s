# Agent Findings

A shared, version-controlled knowledge base for AI agents working on htp-k8s. When you (as any subagent or the orchestrator) learn something non-obvious that a *future* agent would otherwise have to rediscover the hard way — a build/CI gotcha, a dependency constraint, a resolved trip-hazard — record it here. This is the project's "agent memory": it travels with the repo, is read by every dispatched agent, and is visible to the human maintainer.

**Scope:** durable, cross-ticket operational knowledge. Not a changelog (git history covers that), not domain vocabulary (`CONTEXT.md`), not architectural decisions (`docs/adr/`). If a fact belongs in one of those, put it there and link it.

**Upkeep:** keep entries terse and current. If a finding becomes obsolete (e.g. a constraint is lifted), delete or update it rather than leaving stale advice.

## Build & toolchain

- **The app is a single binary.** `task build` builds the frontend (`web/`), copies `web/dist` into `internal/server/dist`, and compiles `bin/htp-k8s` with the frontend embedded via `go:embed`. Running the binary serves the UI plus a `/ws` WebSocket endpoint. There is no separate frontend server to run in production.
- **`go.mod`'s `go` directive is `go 1.26.0`, and CI pins the Go minor explicitly.** We preferred a bare `go 1.26` (minor floor, no forced dev toolchain fetch — see issue #2), but the `k8s.io` v0.36 dependency graph *requires* `go 1.26.0`, so the module graph rejects `go 1.26`. `go 1.26.0` is still only a floor (satisfied by any 1.26.x), so dev machines aren't forced to a specific patch. The catch: `setup-go` with `go-version-file: go.mod` reads that exact `1.26.0` and installs **unpatched** 1.26.0, which makes `govulncheck` flag stdlib vulns fixed in later 1.26.x. So all `setup-go` steps in `build.yml` use `go-version: "1.26"` (minor) + `check-latest: true` to get the newest 1.26.x patch. **Bump that `"1.26"` when the project moves to a new Go minor.**
- **`task test` runs `go test ./cmd/... ./internal/...`, not `./...`** — scoped on purpose. After `npm ci`, `web/node_modules` can contain stray vendored `.go` files (e.g. the `flatted` package) that `./...` would wrongly pull into the module's test graph.

## Wire contract codegen (SceneState → TypeScript, issue #10)

- **The frontend/backend wire contract is Go-first.** `SceneState` (and the `ViewMode` type/values) are defined once in **`internal/scene`** — the single source of truth. The frontend's TypeScript types are **generated** from it into **`web/src/generated/scenestate.ts`** (checked in); never hand-edit that file.
- **Regenerate with `task codegen`** (runs `go tool tygo generate`, config in root `tygo.yaml`). Root `task build` runs `codegen` before `web:build` so the frontend always typechecks against fresh types.
- **Drift gate: `task codegen:verify`** regenerates then `git diff --exit-code -- web/src/generated`, so a Go struct change without a matching regenerate-and-commit fails. It runs in the **Backend (Go)** CI job (a step before `Build`), **not** the Frontend job — tygo needs the Go toolchain, which the Node job lacks.
- **tygo (`github.com/gzuidhof/tygo`) is a pinned Go *tool* dependency** (`go get -tool`, invoked via `go tool tygo`). It correctly stays `// indirect` in `go.mod` after `go mod tidy` (nothing imports it in code) — that's tidy-stable, don't "fix" it.
- **tygo emits `any` for a field whose type lives in another Go package** (e.g. it can't resolve `kube.ViewMode` from a `scene` struct). Keep every type reachable from `SceneState` **co-located in `internal/scene`** so the generated TS is fully typed. This is why `ViewMode` was moved out of `internal/kube` into `internal/scene`.
- **`web/src/generated` is ESLint- and Prettier-ignored** (generated, not hand-styled) — `web/eslint.config.js` `globalIgnores` + `web/.prettierignore`. It is still typechecked by `tsc -b` (tsconfig `include: ["src"]`), so malformed generated TS still fails the frontend build.
- **`/ws` sends a `scene.SceneState` JSON snapshot on connect, then a stream of `scene.SceneDelta` messages (issue #16, ADR-0007).** The first frame is still a bare `SceneState` (no `type` tag — the e2e smoke test and the pre-#17 frontend depend on that); every frame after it is a `SceneDelta` carrying a `type` discriminator (`towerAdded`/`towerRemoved`/`towerMoved`/`panelAdded`/`panelUpdated`/`panelRemoved`) with pointer/omitempty payload fields (tygo emits them as optional, fully typed). `SceneState` shape is unchanged (backward-compatible). Consuming the deltas (the frontend reconciliation reducer) is issue #17; the current frontend ignores them.
- **Scene Deltas are computed by rebuild-and-diff, not event-translation.** `scene.Diff(prev, next SceneState) []SceneDelta` is the pure, deterministic core (fully fake-clientset/unit-testable). `kube.SceneWatcher` (in `internal/kube/watch.go`) drives it: client-go SharedInformers (Pods always + Nodes in Node-mode / Namespaces in Namespace-mode, mirroring `BuildTowers`/`BuildPanels` scoping) coalesce any add/update/delete into a one-slot trigger; a single worker rebuilds the whole `SceneState` via `kube.BuildScene` (the shared snapshot path, so deltas can't drift from snapshots), diffs it, and broadcasts. This is self-correcting (coalesced/missed events still converge) and idempotent. `SnapshotAndSubscribe()` atomically returns the current snapshot + a gap-free delta channel under the broadcaster's lock. A subscriber that overruns its buffer (`subscriberBuffer`) is dropped (channel closed) → its `/ws` connection ends → client reconnects for a fresh snapshot (mirrors k8s "too old resourceVersion → relist"). Grid re-layout: because grid position is a pure function of the sorted tower-name set, adding/removing a Tower can shift others — `towerMoved` deltas carry the new positions so a reducer stays consistent without re-running layout. **Gap (tracked as issue #161):** the OpenShift Project-fallback resource is not watched, so a new Project appears only on the next snapshot/reconnect, not as a live delta — see the residual-gap note on the Panels bullet below for the matching Pods-side limitation and how `Start` now degrades instead of hanging when this bites.
- **The frontend `SceneDelta` discriminated union in `web/src/scene/sceneDelta.ts` is HAND-MAINTAINED — codegen only covers the wire struct.** tygo emits `SceneDelta` as one interface with all fields optional and `SceneDeltaType` as a bare `string` — not the tagged union a reducer wants. So the frontend hand-writes a proper literal-union `SceneDeltaType` + discriminated `SceneDelta` and a `parseSceneDelta(wire)` narrowing adapter at the boundary (validates each kind's required fields; returns `null` + `console.warn` for malformed or unknown/future `type`, never throws). **A new Go delta kind (e.g. `blinkTriggered`, #18) must be mirrored there** — add it to the literal union + a `switch` case (the `never`-typed default makes omission a compile error) and validate its fields; until then `parseSceneDelta` drops it gracefully (the reconnect snapshot recovers full state). Keep the Go `scene.SceneDelta` wire struct as the single tagged struct with optional fields — do NOT try to make tygo emit the union.
- **Panels are nested under their Tower (issue #14): `Tower.Panels []Panel`, no flat scene-level list and no `Panel.tower` FK.** A Panel is always part of exactly one Tower (CONTEXT.md: it sits "on a Tower's face"); `SceneState` stays `{viewMode, towers}`. `kube.BuildPanels` (analogue of `BuildTowers`) lists pods cluster-wide first, derives phase/color, and buckets by owning-Tower name per View Mode via `towerForPod`; `kube.AttachPanels` nests each bucket into its Tower. The server wiring composes `BuildTowers` → `BuildPanels` → `AttachPanels`.
  - **Issue #55: a project-scoped OpenShift user (can list Projects, not pods cluster-wide) now gets real Panels, not an empty set.** `BuildPanels` takes the dynamic client and falls back to `namespaceScopedPods` — bounded-concurrency per-Namespace/Project `Pods.List` calls, same `admittedNamespaceNames` source as `BuildTowers`' Project fallback — only when the cluster-wide list is specifically `apierrors.IsForbidden`. A per-namespace failure that coincides with `ctx` already being done (the fallback's own calls ran into `rebuildTimeout`) fails the whole fallback as an error rather than silently returning a partial pod list.
  - **A degraded rebuild is never published as truth.** `BuildScene` (now `(scene.SceneState, error)`) always returns a usable `SceneState` (ADR-0002) but signals when the Towers or Panels build degraded; `SceneWatcher.rebuildAndBroadcast` treats that error as "don't trust this rebuild" and skips the diff/publish entirely, keeping the last known-good `current` instead of a mass `panelRemoved` wipe — a `namespaceScopedPods` timeout delays the next successful reconcile rather than erasing every client's Panels. `Start`'s own seed still uses a degraded result regardless, since there's no last-known-good yet at cold start. The staleness this trades the wipe for lasts until a rebuild succeeds — one cycle for a transient cause, indefinitely for a persistent one (e.g. no Namespace/Project source at all). Full reasoning lives in `internal/kube/watch.go`'s doc comments (the `rebuild` field, `rebuildAndBroadcast`) — not restated here.
  - **`SceneWatcher.Start` bounds its informer-cache-sync wait instead of blocking forever** when an informer can never sync (e.g. every cluster-scoped informer, for the same project-scoped user #55 targets) — see the `cacheSyncTimeout` const in `watch.go` for the trade-off. A reconnect does NOT re-run `BuildScene`: `SnapshotAndSubscribe` only ever hands out the cached `current`. So this user's server starts and serves a correct initial snapshot but gets no live deltas, ever, until the process restarts. Closing that gap needs a per-Namespace/Project-scoped informer set; tracked as **issue #161**, deliberately out of #55's scope — see `watch.go`'s type doc.

## client-go / kubeconfig resolution

- **`errors.Is(err, fs.ErrPermission)` does NOT reliably match a permission-denied `clientcmd` load failure (issue #128) — despite `errors.Is(err, fs.ErrNotExist)` working fine for the sibling not-found case right next to it. The cause is NOT apimachinery's `errors.aggregate` resisting unwrapping** (an earlier version of this note claimed that; it's wrong — `aggregate` implements `Is(target error) bool` and delegates to `errors.Is` on each contained error, see `k8s.io/apimachinery/pkg/util/errors`' `aggregate.Is`/`aggregate.visit`, so it traverses fine). **The real cause is one layer earlier: `clientcmd`'s own loader discards the typed error before the aggregate is even built.** `LoadFromFile`'s read-failure branch wraps the underlying `*fs.PathError` with `fmt.Errorf("error loading config file %q: %v", filename, err)` — **`%v`, not `%w`** (`k8s.io/client-go@v0.36.2/tools/clientcmd/loader.go:236`, verified against the vendored source) — which flattens it to plain text right there, before `NewAggregate` ever collects it. By the time the error reaches the aggregate, there's nothing typed left to traverse to. (The sibling not-found case never hits this line at all — `os.IsNotExist(err)` short-circuits it a few lines earlier and takes a different, non-flattening path, which is why THAT check "just works" and made the permission one look like it should too.) **Fix: don't try to classify `clientcmd`'s returned error at all — re-open the same file directly and classify *that* error instead** (see `internal/kube/client.go`'s `kubeconfigUnreadable`); only safe/simple when the path being probed is a known, compile-time constant, which it is here (`containerKubeconfigPath`). Don't assume a given `clientcmd`/client-go error wraps cleanly with `%w` all the way down — check the actual source (`grep` the vendored module under `$(go list -m -f '{{.Dir}}' k8s.io/client-go)`) before relying on `errors.Is`/`errors.As` against it.

## CI & security scanning

- **`govulncheck ./...` is a blocking PR check** (Backend job). It fails the build on any *reachable* vulnerability. This has bitten PRs twice; treat it as a first-class gate, and run `govulncheck ./...` locally before pushing.
- **Keep `golang.org/x/net` at ≥ v0.57.0.** It arrives transitively via the kind/KWOK/k8s client libraries. v0.38.0 shipped two reachable vulns (GO-2026-5026 idna, GO-2026-4918 http2). When you `go mod tidy`, verify x/net did not get downgraded below v0.57.0 — a botched go.sum merge can silently drop it.
- **`npm audit` is a blocking PR check** (Frontend job). Run it locally before pushing if you touched `web/` dependencies.
- **Branch protection on `main`** (ruleset id 18831212): PRs required (0 approvals), **strict** required-status-checks policy (branch must be up to date with `main` before merge), with **all five** PR checks required — `Backend (Go)`, `Frontend (Node)`, `Backend integration (kind + KWOK)`, `E2E (Playwright)`, `Container scan (Trivy)`. Direct pushes to `main` are rejected — always go through a PR. (The three heavy jobs were made required by the owner after #70; earlier notes calling them "not yet required" are stale.)
- **A PR opened *before* a workflow existed on `main` won't have run the checks.** Update the PR branch with `main` (`gh pr update-branch <n>`) to produce the commit event that triggers them.
- **Two-tier PR jobs (issue #8).** On top of the fast `Backend (Go)` / `Frontend (Node)` jobs, `build.yml` has three heavier jobs, each separate so the fast tier stays fast: `Backend integration (kind + KWOK)`, `E2E (Playwright)`, `Container scan (Trivy)`. All three are now **required** status checks on `main` (see the branch-protection note above).
- **Docs-only PRs skip the three heavy jobs (issue #76), and STILL merge.** `build.yml` has a cheap `changes` gate job (`dorny/paths-filter`, SHA-pinned) that outputs `nondocs`; `Backend integration (kind + KWOK)`, `E2E (Playwright)`, and `Container scan (Trivy)` carry `needs: changes` + `if: needs.changes.outputs.nondocs == 'true'`, so a PR whose every changed path is under `**/*.md`, `docs/**`, or `LICENSE` skips ~10-15 min of cluster-spinning. **The whole point — never break with `on.pull_request.paths-ignore`:** a *skipped workflow* leaves the five required checks pending forever → a docs PR becomes permanently un-mergeable. The workflow still triggers on every PR; only the heavy *jobs* skip at the job level, and a **job-level-skipped required check counts as passing** for the ruleset (verified on a docs-only test PR for #76 — the three heavy checks showed `skipped` and `mergeStateStatus` was not BLOCKED on them; no shim job was needed). Precise definition uses `predicate-quantifier: 'every'` (`**` AND `!**/*.md` AND `!docs/**` AND `!LICENSE`), so **anything** non-docs — code, `web/`, config, and especially `.github/**`, plus any mixed docs+code PR — flips `nondocs` true and runs the full suite (a `.github` workflow change must never skip its own validation). Don't rewrite the filter as an OR of bare `!` patterns; that would mis-classify docs files. The nightly full-suite backstop (#29) still runs the heavy jobs against live vuln DBs regardless of paths, so a docs-only stretch can't hide a newly-published CVE.
- **`build.yml` (renamed from `pr.yml`, issue #95) runs on BOTH `pull_request` and `push: branches:[main]`, reusing the same jobs.** This gives every `main` commit its own check-runs — the prerequisite for the `release.yml` green-commit guard below, which reads exactly those check-runs by SHA. The docs-skip `changes` gate (previous bullet) had to keep working for both events: on `push`, `dorny/paths-filter` can't use the PR-diff API, so the `changes` job conditionally checks out the repo (`fetch-depth: 0`) and sets `base: main` — since `main` IS the branch just pushed to, this puts the action in its "long-lived branch" mode (diff against the commit on `main` immediately before the push), which correctly isolates the merged PR's own file set. A docs-only push to `main` still makes the three heavy jobs report `skipped`, which the green-commit guard (like branch protection) treats as passing. The `concurrency.group` is keyed on `github.event.pull_request.number || github.ref` so PR runs, and push-to-main runs, never share a cancellation group.
- **`release.yml` has a green-commit guard job (issue #95) gating every other release job via `needs:`.** It polls `gh api repos/<repo>/commits/<tag-sha>/check-runs` (bounded retry loop, not a single read — the tag can be pushed moments after the merge, before `main`'s CI has finished) and requires every required-check name to be `completed` with conclusion `success` **or** `skipped` (mirrors branch protection's own skipped-counts-as-passing rule); any `failure`/`cancelled`/`timed_out` fails fast without waiting out the timeout, and a check that never appears/completes fails once the poll deadline is hit. Needs only `checks: read` (+ default `contents: read`) — deliberately NOT branch-protection-settings read (that needs higher privilege than the default `GITHUB_TOKEN` should be granted). This is what makes tagging an untested or off-`main` commit fail loudly instead of silently building it.
- **Release Trivy scans BOTH per-platform child images, not the multi-arch index (issue #95).** Scanning the index resolves to whatever arch the runner defaults to (amd64) — arm64 was shipped unscanned before this. The `image` job's `ko build --image-refs=<file>` output (line 1 = index, subsequent lines = per-platform children) is parsed into a JSON array of child digests (`needs.image.outputs.children`); a `trivy-scan` matrix job (mirroring the `attest-sboms` fan-out pattern) scans each child by digest with a distinct SARIF `category` per child (keyed off the digest, not a hardcoded platform label — don't assume image-refs line order maps to a specific arch). Same non-blocking-findings / blocking-scan-failure semantics as before.
- **Release attestation now covers the index AND each per-platform child (issue #95).** A new matrix job (again mirroring `attest-sboms`) attests each child digest from `needs.image.outputs.children` with `actions/attest-build-provenance` + `push-to-registry: true`, alongside the pre-existing index attestation. The platform list is never hardcoded anywhere in the attestation path — everything fans out from the dynamic `children` array, so adding/removing a `ko build --platform` entry doesn't require touching the attestation job.
- **Two recurring CI transients are now mitigated (issue #66) — don't just blind-re-run these anymore; a repeat means the fix regressed, so investigate.** (1) *`arduino/setup-task` rate-limit:* every `Set up Task` step in **both `build.yml` and `release.yml`** now passes `repo-token: ${{ secrets.GITHUB_TOKEN }}`, so the go-task release lookup uses the authenticated GitHub API rate limit instead of the low anonymous one that used to fail the step before any real work ran. Every `setup-task` step must keep this (no extra permissions — `GITHUB_TOKEN` is available under the workflow's `contents: read`). (2) *Playwright install hang:* the E2E job's `Install Playwright browser + system deps` step (`npx playwright install --with-deps chromium`) intermittently hung (~19 min observed); it now has a step `timeout-minutes` plus a 3-attempt retry loop with a per-attempt `timeout 240` — a single hang is killed fast and retried, but the job still fails (`exit 1`) if all attempts are exhausted, so a *persistent* failure is never masked.
- **The PR-time Trivy container scan: CVE findings are non-blocking, but a scan *operational* failure fails the `Container scan (Trivy)` check (issue #70, matching release semantics).** It builds a throwaway local image with `ko build --local` (no registry push; `KO_DOCKER_REPO=ko.local`), then Trivy emits SARIF that is uploaded to the Security tab. The job needs a job-level `permissions: security-events: write` (the workflow default is only `contents: read`) or the SARIF upload 403s. Semantics (mirrors `release.yml` since #69/#70): keep `exit-code: "0"` so CVE **findings** exit 0 and never fail the PR, but the scan step has **NO `continue-on-error`**, so a genuine scan *operation* failure (auth/401, image-pull, DB download, hang) exits non-zero and turns the check RED — catch a broken scan in the PR rather than at release. The SARIF upload likewise has **no `continue-on-error` and no `if: always()`** (both removed in #70), so it runs only after a successful scan and a failed upload fails the job — a real SARIF reaches the Security tab or the check goes red. **The scan step has a step-level `timeout-minutes` (#70),** so a setup hang (the trivy binary/DB download stall seen on #67) fails fast instead of stalling to the job timeout — and, with `continue-on-error` gone, that timeout correctly reds the check. trivy-action is a `uses:` step so a shell retry loop can't wrap it directly; the timeout is the guard (same flake class as the Playwright-install timeout in #66/#68). NOTE: making `Container scan (Trivy)` a *required* status check is a separate branch-protection-ruleset change the maintainer/orchestrator flips **after** #70 lands (order matters — requiring it before the timeout existed would risk a hang stalling a now-required check, the #67 situation).
- **`ko build --local ./cmd/htp-k8s` works with only the committed placeholder `internal/server/dist/index.html`** — `go:embed all:dist` matches it, so ko can compile the binary into an image for CVE scanning without first running the frontend build. (The real ko/GoReleaser release build is issue #30.)
- **Release-time supply-chain steps live in `release.yml` and only run on a `v*` tag push (issues #30/#31, ADR-0005).** The release Trivy scan scans the *released* image by digest (not a throwaway `--local` build) and draws a sharper line than the PR one: `exit-code: "0"` keeps CVE **findings** non-blocking, but the scan step has **NO `continue-on-error`**, so a scan *operational* failure (auth/401, image-pull, DB download, hang) fails the release; the SARIF upload likewise has no `continue-on-error`/`if: always()`, so a real SARIF reaches the Security tab or the build fails. The scan step also has a step-level `timeout-minutes` (#70) so a hang fails fast — and, with `continue-on-error` gone, that timeout correctly fails the release. (The PR-time scan now uses the **same** fail-on-scan-operation-error + timeout semantics as of #69/#70 — see the PR-Trivy note above; it is no longer fully report-only.) GitHub-native attestation is split by artifact type because the actions constrain how subjects are passed: (1) **binaries** get one `actions/attest-build-provenance` call with `subject-checksums: dist/checksums.txt` (GoReleaser's sha256 manifest is the whole subject list in one shot); (2) the **container image** gets `attest-build-provenance` with `subject-name`/`subject-digest` + `push-to-registry: true` — set `subject-name` to the **bare `KO_DOCKER_REPO`** (`ghcr.io/<owner>/htp-k8s`, NO tag — the action rejects a tagged name) and capture the digest via **`ko build --image-refs=<file>`** (reads the digest-pinned `repo@sha256:...` from that file), NOT by string-parsing ko's stdout. Parsing stdout was fragile — ko can print a `repo:tag` ref instead of `repo@digest`, and the first release dry-run (`v0.1.0-rc.1`, #92) fed a *tagged* name to the attestation → `Error: Invalid image name`. See #92; (3) **SBOMs** need one `actions/attest-sbom` call *per archive* (the action binds ONE sbom-path to its subject per invocation, and GoReleaser emits one CycloneDX SBOM per archive), so they fan out in a `matrix` job that consumes the archives+SBOMs via an `upload-artifact`/`download-artifact` handoff (a job can't matrix over files it doesn't yet have). **Job-level permissions (least privilege, not workflow-wide):** attestation needs `id-token: write` + `attestations: write`; the image job also keeps `packages: write` (GHCR push + `push-to-registry` attestation) and adds `security-events: write` for the Trivy SARIF upload. Verify all this statically with `actionlint` — `release.yml` is tag-triggered so it never runs on the PR that changes it.

## Testing

- **Frontend fixtures go through `web/src/test-support/sceneFixtures.ts` (issue #15).** `makeSceneState`/`makeTower`/`makePanel` each return a COMPLETE, valid wire-contract object with all required fields defaulted and take a `Partial<>` of overrides. Build `SceneState`/`Tower`/`Panel` test data through these, not hand-written literals — a new required field on the Go contract (this bit us 3×: towers-required, panels-required, panels-nested) is then a one-line change here, not a sweep across N test files. The pure `SceneState`→instance layout seam (`web/src/scene/panelLayout.ts`, `panelInstances`/`resolvePanel`) is unit-tested WebGL-free; the `<instancedMesh>` renderer (`web/src/scene/Panels.tsx`) is thin glue validated by the Playwright screenshot (ADR-0004), and is `vi.mock`-stubbed in `Scene.test.tsx` since jsdom has no WebGL.
- **A green e2e is NOT a verified visual — look at the hero screenshot.** Playwright's `toBeVisible()` / DOM asserts pass for an element that is present but *outside the camera frame*, so a UI feature can be "green" while its screenshot shows nothing. This hid real defects twice: #80's Detail Popup rendered off-frame (the in-world drei `<Html>` was anchored at a Panel the camera wasn't looking at — `toBeVisible` still passed), and #84's Demo Mode flew into empty space. **Verify the named hero screenshot actually shows the feature *framed* before calling a visual change done**, and have the e2e take an explicit `page.screenshot()` while the feature is on screen (the auto `test-finished` shot fires at teardown, often after the feature is dismissed). For in-world `<Html>` popups specifically: the popup is anchored at the clicked element's world point, so it only frames on screen if the camera is looking there — drive the *same Focus fly-to a real click does* (the detail test hook mirrors the full click: `requestFocus` + `select`), not selection alone.
- **Experiential features (camera choreography, animation, aesthetic "feel") need a live human review — CI proves mechanics, not feel (ADR-0004).** #84 Demo Mode passed e2e (toggle works, camera moves, banks, hands back jump-free) yet the *route* was wrong — it flew above/around the Towers into the void rather than among them — and only a manual run surfaced it. For such tickets, plan the manual checkpoint and flag in the PR that the feel needs a live look *before* merge-readiness, rather than discovering it at review time.
- **Building/calibrating a layer-1 motion-quality invariant is its own loop with its own failure modes, separate from the motion bug it guards** — see ADR-0011's amendment (#105 iterations 1-4) for the rules and the failures that earned them. Not restated here or in the ADR's bounds catalogue (`web/src/scene/demoModeSmoothness.test.ts`) — one home, to avoid drift.
- **Running the app locally against a populated multi-tower scene** (for manual/explorative testing): `kind create cluster` → `./test/e2e/kwok/seed.sh` (1 real + 6 KWOK nodes = 7 Towers, 30 pods across a phase mix) → `. ~/.profile && task build` → `./bin/htp-k8s` (serves UI + `/ws` + `/api` on `127.0.0.1:8080` — loopback only by default since issue #127, which is fine for `http://localhost:8080`; the `/ws` URL is baked to `localhost:8080`, so keep that port unless you rebuild with `VITE_WS_URL`). The binary fails startup without a reachable cluster (ADR-0009 / #9). Full recipe + gotchas + teardown in issue #90.
- **The version evidence behind ADR-0012's "no config-level way to set a container-runtime env var" claim (issue #127) lives here, not in the ADR — #137 tracks giving the ADR itself a version pin; source it from this bullet.** Checked against ko v0.19.1 and GoReleaser v2.17.0 source: `internal/pipe/ko/ko.go` feeds GoReleaser's `kos[].env` straight into the same `build.Config.Env` plain ko's own `.ko.yaml` `builds[].env` uses (both Go **build**-time env only), and ko itself only ever appends to an image's runtime `Config.Env` for two hardcoded internal values, `KO_DATA_PATH`/`PATH` — no exposed flag or config field for anything else. Full reasoning (why this stays a documented `-e HTP_K8S_ADDR=:8080` flag rather than baked into the image, the rejected image-only ldflag alternative, the `--network host` fail-open trade-off): [ADR-0012](../adr/0012-secure-by-default-network-binding.md) — not restated here, to avoid the two copies drifting.
- **For a container reaching a *local* kind cluster, don't use `--network host` (issue #127).** It shares the host's network namespace outright, so Docker's `-p` publish mapping is silently ignored — a container bound to `:8080` under `--network host` is bound to `:8080` on the *host*, across every interface, regardless of what `-p` says. Verified empirically (real kind cluster, real ko-built image): join kind's own `kind` Docker network instead (`docker run --network kind ...`) and mount `kind get kubeconfig --internal --name <cluster>` (which resolves `https://<cluster>-control-plane:6443` from inside that network) rather than the default externally-facing kubeconfig (`https://127.0.0.1:<port>`, meaningless inside a bridge-networked container). With that, ordinary bridge networking + `-p 127.0.0.1:8080:8080` works end-to-end, confirmed via `ss -ltn` (host listener really is `127.0.0.1:<port>`, not `0.0.0.0`) and a failed connection attempt from the host's real LAN IP.
- **Integration tests are gated behind the `//go:build integration` build tag** and are therefore *not* run by the fast PR workflow's `task test` step (that's by design — the fast jobs stay fast). Run them locally with `go test -tags=integration ./internal/...`. As of issue #8 they run in CI as their own `Backend integration (kind + KWOK)` job (`go test -tags=integration ./cmd/... ./internal/...` — same `./cmd/... ./internal/...` scoping as `task test`, to dodge the `web/node_modules` stray-`.go` hazard). Do **not** add `-tags=integration` to the fast Backend `Test` step — that would force every PR's fast job to spin up kind+KWOK.
- **The Playwright e2e CI job must install the browser with `npx playwright install --with-deps chromium`** — the `--with-deps` apt step supplies system libraries a fresh Ubuntu runner lacks. The `web:e2e` task itself only runs `playwright install chromium` (download, no system deps), so the CI job runs the `--with-deps` install before `task web:e2e`. That job also needs Node + Go + go-task all on PATH, because Playwright's `webServer` runs `task build` to produce and launch the real binary. Screenshots/video/traces land in `web/e2e-results/` (uploaded as the `playwright-e2e-results` artifact; HTML report in `web/playwright-report/`). **That install step is now hardened against a runner-transient hang (issue #66):** it has a step `timeout-minutes` plus a bounded retry loop with a per-attempt `timeout 240`, so a stuck install fails fast and auto-recovers instead of burning the runner to the job timeout — the job still fails if all attempts are exhausted. See the CI-hardening note above.
- **The kind + KWOK harness lives at `internal/testcluster`** (`testcluster.New(t, ...)`, from issue #5). It programmatically spins up a real single-node kind cluster and attaches a KWOK controller — **Docker is the only prerequisite** (no `kind`/`kwok` CLI, no pre-existing cluster). It ran live in ~45s in the build environment. Teardown is registered via `t.Cleanup` before creation, so clusters are always removed even on panic/failure. Use `AddFakeNodes`/`AddFakePods` for scale without real containers.
- **KWOK manifests are vendored** from pinned upstream v0.8.0 under `internal/testcluster/manifests/` (with attribution). Don't hand-edit them casually; re-vendor from a pinned upstream tag if they need updating.
- **The fake clientset (`k8s.io/client-go/kubernetes/fake`) drives SharedInformers fine** — creating/updating/deleting objects through it delivers watch events to informers. That's how `SceneWatcher`'s "watch events → Scene Deltas" unit tests (`internal/kube/watch_test.go`) run without a real cluster. Use a timeout-bounded channel read for the async informer→worker→broadcast hop (a few seconds is plenty).
- **For deterministic real-cluster pod-lifecycle deltas, bind the pod to a *bare* Node (no KWOK annotation, no kubelet) and drive its status yourself.** The `SceneWatcher` integration test (`watch_integration_test.go`) needs a *separable* create → update → delete so each yields its own delta; a KWOK-managed node flips Pending→Running so fast it **coalesces the create+update into a single `panelAdded`** (rebuild-and-diff reports net change), so there's no distinct `panelUpdated`. On a bare node nothing races you: create → `panelAdded`, then `UpdateStatus` phase=Running → `panelUpdated`. **Two gotchas of bare (kubelet-less) nodes:** a bound pod stays `Pending` until you set its status (fine — the panel appears regardless of phase), and **`Delete` hangs in `Terminating` forever** (no kubelet completes graceful deletion) — force-delete with `GracePeriodSeconds: 0` so `panelRemoved` fires. Assert by reconstructing the scene (snapshot + applied deltas) and waiting for convergence, not on exact per-event deltas (a real watch coalesces/races).

## Writing docs

- **Verify a number once, then re-verify every *second* narration of it — the copy drifts, not the original.** While writing #136 the cited figures were checked against git history and were correct; a retelling of the same facts in a second file overstated them anyway, and review caught it in the copy, not the source. Verifying a claim does not inoculate the paraphrase you write two files later. When a fact is needed in two places, link to the file that owns it instead of retelling it — see `docs/agents/documentation-proportionality.md`.

## Parallel-dispatch hazards

- **Two tickets that both edit `go.mod`/`go.sum` will conflict when the second merges**, even with no source overlap. This happened with PRs #35 (added gorilla/websocket) and #36 (added kind/KWOK deps). Resolution: merge `main` into the lagging branch, take one side's `go.mod`/`go.sum` wholesale, re-run `go mod tidy`, then **re-verify `govulncheck` and the x/net floor survived**. Prefer scoping parallel tickets so only one touches Go dependencies at a time.
- **File-ownership boundaries prevent most conflicts.** When dispatching parallel work, give each agent an explicit disjoint file scope (backend → `cmd/`+`internal/`, frontend → `web/`, CI → `.github/`, etc.) and name the sibling ticket's scope so it stays clear.
- **When parallel PRs *must* both touch a shared composition root (e.g. `web/src/Scene.tsx`), scope each to a *minimal, additive* edit in a *different region* — git then auto-merges them with no manual reconcile.** Verified: #83 (mount `<FloorLanes>`) and #84 (add a demo toggle + a `<FreeFlyControls demoActive>` prop) both edited `Scene.tsx` + `Scene.test.tsx`, kept their edits to small isolated insertions, and `gh pr update-branch` auto-combined them correctly (no conflict). Instruct each agent explicitly to keep the shared-root edit minimal and NOT restructure it.
- **"Disjoint" is a claim to verify by inspection, not assume.** #80 and #81 were dispatched as disjoint but both edited the *same region* of `web/src/scene/Panels.tsx` (the imports + the same hook), so they conflicted on merge and needed a hand reconcile. Before dispatching parallel work — and before claiming two PRs won't collide — actually list each ticket's likely/real changed files (`git diff --name-only <base> <branch>`) and confirm the overlap.
- **`.claude/agents/*.md` custom subagent types may not be available in every runtime.** When they aren't, dispatch a `general-purpose` agent and fold the role's persona (from the corresponding `.claude/agents/*.md`) into the prompt instead.
- **Close issues on PR *merge*, not PR *open*.** Native issue dependencies count *open* blockers, so closing an issue when its PR opens makes downstream tickets read as unblocked before their dependency's code is on `main`. Use `Closes #<n>` in the PR body so GitHub auto-closes on merge; the orchestrator gates the frontier on "blocker PR merged," not "issue closed." See `docs/agents/issue-tracker.md`.
- **The e2e (Playwright) job must provision a real cluster.** Since issue #9 the binary *fails startup* when it cannot reach a Kubernetes cluster. Playwright's `webServer` launches that binary, so the e2e job now creates a single-node `kind` cluster (`helm/kind-action`, pinned to SHA) and exports `KUBECONFIG` to `~/.kube/config`; `web/playwright.config.ts`'s `webServer.env` passes `KUBECONFIG` through so the launched binary connects (client-go honors `KUBECONFIG` / `~/.kube/config`). A fresh kind cluster makes the CI user cluster-admin, so the app detects Node view mode. Do NOT weaken the fail-on-unreachable startup to dodge this — provisioning a cluster is the intended fix (ADR-0004). The smoke test asserts a *well-formed* `/ws` frame (parsed JSON, not a frozen string) so it survives the wire format growing: as of issue #10 that frame is a `SceneState` snapshot `{viewMode:"node"|"namespace"}` (no `type` field — the ad-hoc `{type:"viewMode",...}` message was replaced), and the test resolves on the first frame with a valid `viewMode` without over-constraining the rest.
- **Two individually-green PRs can conflict *semantically* once both land on `main`** — not just via file/`go.mod` overlap. The e2e harness (issue #7) launched `./bin/htp-k8s` back when it was a clusterless placeholder; issue #9 later made the binary fail startup without a reachable cluster. Each PR was green alone, but together they broke the e2e check on every PR (a docs-only PR, #46, first surfaced it). Lesson: when app behavior changes in a way that affects *how it is launched* (startup preconditions, required env, ports, config), check and update the e2e harness (`web/playwright.config.ts`, `.github/workflows/build.yml` e2e job) in the same change.
