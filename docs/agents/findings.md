# Agent Findings

A shared, version-controlled knowledge base for AI agents working on htp-k8s. When you (as any subagent or the orchestrator) learn something non-obvious that a *future* agent would otherwise have to rediscover the hard way — a build/CI gotcha, a dependency constraint, a resolved trip-hazard — record it here. This is the project's "agent memory": it travels with the repo, is read by every dispatched agent, and is visible to the human maintainer.

**Scope:** durable, cross-ticket operational knowledge. Not a changelog (git history covers that), not domain vocabulary (`CONTEXT.md`), not architectural decisions (`docs/adr/`). If a fact belongs in one of those, put it there and link it.

**Upkeep:** keep entries terse and current. If a finding becomes obsolete (e.g. a constraint is lifted), delete or update it rather than leaving stale advice.

## Build & toolchain

- **The app is a single binary.** `task build` builds the frontend (`web/`), copies `web/dist` into `internal/server/dist`, and compiles `bin/htp-k8s` with the frontend embedded via `go:embed`. Running the binary serves the UI plus a `/ws` WebSocket endpoint. There is no separate frontend server to run in production.
- **`go.mod`'s `go` directive is `go 1.26.0`, and CI pins the Go minor explicitly.** We preferred a bare `go 1.26` (minor floor, no forced dev toolchain fetch — see issue #2), but the `k8s.io` v0.36 dependency graph *requires* `go 1.26.0`, so the module graph rejects `go 1.26`. `go 1.26.0` is still only a floor (satisfied by any 1.26.x), so dev machines aren't forced to a specific patch. The catch: `setup-go` with `go-version-file: go.mod` reads that exact `1.26.0` and installs **unpatched** 1.26.0, which makes `govulncheck` flag stdlib vulns fixed in later 1.26.x. So all `setup-go` steps in `pr.yml` use `go-version: "1.26"` (minor) + `check-latest: true` to get the newest 1.26.x patch. **Bump that `"1.26"` when the project moves to a new Go minor.**
- **`task test` runs `go test ./cmd/... ./internal/...`, not `./...`** — scoped on purpose. After `npm ci`, `web/node_modules` can contain stray vendored `.go` files (e.g. the `flatted` package) that `./...` would wrongly pull into the module's test graph.

## Wire contract codegen (SceneState → TypeScript, issue #10)

- **The frontend/backend wire contract is Go-first.** `SceneState` (and the `ViewMode` type/values) are defined once in **`internal/scene`** — the single source of truth. The frontend's TypeScript types are **generated** from it into **`web/src/generated/scenestate.ts`** (checked in); never hand-edit that file.
- **Regenerate with `task codegen`** (runs `go tool tygo generate`, config in root `tygo.yaml`). Root `task build` runs `codegen` before `web:build` so the frontend always typechecks against fresh types.
- **Drift gate: `task codegen:verify`** regenerates then `git diff --exit-code -- web/src/generated`, so a Go struct change without a matching regenerate-and-commit fails. It runs in the **Backend (Go)** CI job (a step before `Build`), **not** the Frontend job — tygo needs the Go toolchain, which the Node job lacks.
- **tygo (`github.com/gzuidhof/tygo`) is a pinned Go *tool* dependency** (`go get -tool`, invoked via `go tool tygo`). It correctly stays `// indirect` in `go.mod` after `go mod tidy` (nothing imports it in code) — that's tidy-stable, don't "fix" it.
- **tygo emits `any` for a field whose type lives in another Go package** (e.g. it can't resolve `kube.ViewMode` from a `scene` struct). Keep every type reachable from `SceneState` **co-located in `internal/scene`** so the generated TS is fully typed. This is why `ViewMode` was moved out of `internal/kube` into `internal/scene`.
- **`web/src/generated` is ESLint- and Prettier-ignored** (generated, not hand-styled) — `web/eslint.config.js` `globalIgnores` + `web/.prettierignore`. It is still typechecked by `tsc -b` (tsconfig `include: ["src"]`), so malformed generated TS still fails the frontend build.
- **`/ws` sends a `scene.SceneState` JSON snapshot on connect, then a stream of `scene.SceneDelta` messages (issue #16, ADR-0007).** The first frame is still a bare `SceneState` (no `type` tag — the e2e smoke test and the pre-#17 frontend depend on that); every frame after it is a `SceneDelta` carrying a `type` discriminator (`towerAdded`/`towerRemoved`/`towerMoved`/`panelAdded`/`panelUpdated`/`panelRemoved`) with pointer/omitempty payload fields (tygo emits them as optional, fully typed). `SceneState` shape is unchanged (backward-compatible). Consuming the deltas (the frontend reconciliation reducer) is issue #17; the current frontend ignores them.
- **Scene Deltas are computed by rebuild-and-diff, not event-translation.** `scene.Diff(prev, next SceneState) []SceneDelta` is the pure, deterministic core (fully fake-clientset/unit-testable). `kube.SceneWatcher` (in `internal/kube/watch.go`) drives it: client-go SharedInformers (Pods always + Nodes in Node-mode / Namespaces in Namespace-mode, mirroring `BuildTowers`/`BuildPanels` scoping) coalesce any add/update/delete into a one-slot trigger; a single worker rebuilds the whole `SceneState` via `kube.BuildScene` (the shared snapshot path, so deltas can't drift from snapshots), diffs it, and broadcasts. This is self-correcting (coalesced/missed events still converge) and idempotent. `SnapshotAndSubscribe()` atomically returns the current snapshot + a gap-free delta channel under the broadcaster's lock. A subscriber that overruns its buffer (`subscriberBuffer`) is dropped (channel closed) → its `/ws` connection ends → client reconnects for a fresh snapshot (mirrors k8s "too old resourceVersion → relist"). Grid re-layout: because grid position is a pure function of the sorted tower-name set, adding/removing a Tower can shift others — `towerMoved` deltas carry the new positions so a reducer stays consistent without re-running layout. **Gap (watch-side analogue of #55):** the OpenShift Project-fallback resource is not watched, so a new Project appears only on the next snapshot/reconnect, not as a live delta.
- **The frontend `SceneDelta` discriminated union in `web/src/scene/sceneDelta.ts` is HAND-MAINTAINED — codegen only covers the wire struct.** tygo emits `SceneDelta` as one interface with all fields optional and `SceneDeltaType` as a bare `string` — not the tagged union a reducer wants. So the frontend hand-writes a proper literal-union `SceneDeltaType` + discriminated `SceneDelta` and a `parseSceneDelta(wire)` narrowing adapter at the boundary (validates each kind's required fields; returns `null` + `console.warn` for malformed or unknown/future `type`, never throws). **A new Go delta kind (e.g. `blinkTriggered`, #18) must be mirrored there** — add it to the literal union + a `switch` case (the `never`-typed default makes omission a compile error) and validate its fields; until then `parseSceneDelta` drops it gracefully (the reconnect snapshot recovers full state). Keep the Go `scene.SceneDelta` wire struct as the single tagged struct with optional fields — do NOT try to make tygo emit the union.
- **Panels are nested under their Tower (issue #14): `Tower.Panels []Panel`, no flat scene-level list and no `Panel.tower` FK.** A Panel is always part of exactly one Tower (CONTEXT.md: it sits "on a Tower's face"), so nesting makes orphan panels structurally impossible and gives the frontend the tower's position for free; `SceneState` stays `{viewMode, towers}`. `kube.BuildPanels` is the Pod→Panel seam (analogue of `BuildTowers`): it lists pods cluster-wide, derives a phase-like status (`scene.PodPhase`, incl. CrashLoopBackOff from container waiting reasons) and its color (`scene.ColorForPhase`), and **buckets** panels by owning-Tower name **per View Mode** (Node in Node-mode, Namespace in Namespace-mode) via `towerForPod` — the sole mode-dependent spot, so panels re-nest on a mode switch. `kube.AttachPanels(towers, byTower)` then nests each bucket into its Tower (empty non-nil slice for a Tower with no pods; a bucket matching no Tower is dropped). The server wiring composes `BuildTowers` → `BuildPanels` → `AttachPanels`. **Gap left for issue #55:** unlike `BuildTowers` (which has an OpenShift Project fallback), `BuildPanels` has no per-Project pod fallback, so a restricted OpenShift user who can list Projects but not pods cluster-wide gets Towers but empty Panels (degrades gracefully per ADR-0002, doesn't hard-fail).

## CI & security scanning

- **`govulncheck ./...` is a blocking PR check** (Backend job). It fails the build on any *reachable* vulnerability. This has bitten PRs twice; treat it as a first-class gate, and run `govulncheck ./...` locally before pushing.
- **Keep `golang.org/x/net` at ≥ v0.57.0.** It arrives transitively via the kind/KWOK/k8s client libraries. v0.38.0 shipped two reachable vulns (GO-2026-5026 idna, GO-2026-4918 http2). When you `go mod tidy`, verify x/net did not get downgraded below v0.57.0 — a botched go.sum merge can silently drop it.
- **`npm audit` is a blocking PR check** (Frontend job). Run it locally before pushing if you touched `web/` dependencies.
- **Branch protection on `main`** (ruleset id 18831212): PRs required (0 approvals), with required status checks `Backend (Go)` and `Frontend (Node)`. Direct pushes to `main` are rejected — always go through a PR.
- **A PR opened *before* a workflow existed on `main` won't have run the checks.** Update the PR branch with `main` (`gh pr update-branch <n>`) to produce the commit event that triggers them.
- **Two-tier PR jobs (issue #8).** On top of the fast `Backend (Go)` / `Frontend (Node)` jobs, `pr.yml` has three heavier jobs, each separate so the fast tier stays fast: `Backend integration (kind + KWOK)`, `E2E (Playwright)`, `Container scan (Trivy)`. **They are NOT (yet) required status checks** — making them required is a branch-protection-ruleset change left to the repo owner.
- **Two recurring CI transients are now mitigated (issue #66) — don't just blind-re-run these anymore; a repeat means the fix regressed, so investigate.** (1) *`arduino/setup-task` rate-limit:* every `Set up Task` step in **both `pr.yml` and `release.yml`** now passes `repo-token: ${{ secrets.GITHUB_TOKEN }}`, so the go-task release lookup uses the authenticated GitHub API rate limit instead of the low anonymous one that used to fail the step before any real work ran. Every `setup-task` step must keep this (no extra permissions — `GITHUB_TOKEN` is available under the workflow's `contents: read`). (2) *Playwright install hang:* the E2E job's `Install Playwright browser + system deps` step (`npx playwright install --with-deps chromium`) intermittently hung (~19 min observed); it now has a step `timeout-minutes` plus a 3-attempt retry loop with a per-attempt `timeout 240` — a single hang is killed fast and retried, but the job still fails (`exit 1`) if all attempts are exhausted, so a *persistent* failure is never masked.
- **The Trivy container scan is report-only, never blocking** (ADR-0005). It builds a throwaway local image with `ko build --local` (no registry push; `KO_DOCKER_REPO=ko.local`), then Trivy emits SARIF that is uploaded to the Security tab. The job needs a job-level `permissions: security-events: write` (the workflow default is only `contents: read`) or the SARIF upload 403s. Keep `exit-code: "0"` + `continue-on-error` on the scan/upload steps so a scanner hiccup or finding never fails the PR.
- **`ko build --local ./cmd/htp-k8s` works with only the committed placeholder `internal/server/dist/index.html`** — `go:embed all:dist` matches it, so ko can compile the binary into an image for CVE scanning without first running the frontend build. (The real ko/GoReleaser release build is issue #30.)

## Testing

- **Frontend fixtures go through `web/src/test-support/sceneFixtures.ts` (issue #15).** `makeSceneState`/`makeTower`/`makePanel` each return a COMPLETE, valid wire-contract object with all required fields defaulted and take a `Partial<>` of overrides. Build `SceneState`/`Tower`/`Panel` test data through these, not hand-written literals — a new required field on the Go contract (this bit us 3×: towers-required, panels-required, panels-nested) is then a one-line change here, not a sweep across N test files. The pure `SceneState`→instance layout seam (`web/src/scene/panelLayout.ts`, `panelInstances`/`resolvePanel`) is unit-tested WebGL-free; the `<instancedMesh>` renderer (`web/src/scene/Panels.tsx`) is thin glue validated by the Playwright screenshot (ADR-0004), and is `vi.mock`-stubbed in `Scene.test.tsx` since jsdom has no WebGL.
- **Integration tests are gated behind the `//go:build integration` build tag** and are therefore *not* run by the fast PR workflow's `task test` step (that's by design — the fast jobs stay fast). Run them locally with `go test -tags=integration ./internal/...`. As of issue #8 they run in CI as their own `Backend integration (kind + KWOK)` job (`go test -tags=integration ./cmd/... ./internal/...` — same `./cmd/... ./internal/...` scoping as `task test`, to dodge the `web/node_modules` stray-`.go` hazard). Do **not** add `-tags=integration` to the fast Backend `Test` step — that would force every PR's fast job to spin up kind+KWOK.
- **The Playwright e2e CI job must install the browser with `npx playwright install --with-deps chromium`** — the `--with-deps` apt step supplies system libraries a fresh Ubuntu runner lacks. The `web:e2e` task itself only runs `playwright install chromium` (download, no system deps), so the CI job runs the `--with-deps` install before `task web:e2e`. That job also needs Node + Go + go-task all on PATH, because Playwright's `webServer` runs `task build` to produce and launch the real binary. Screenshots/video/traces land in `web/e2e-results/` (uploaded as the `playwright-e2e-results` artifact; HTML report in `web/playwright-report/`). **That install step is now hardened against a runner-transient hang (issue #66):** it has a step `timeout-minutes` plus a bounded retry loop with a per-attempt `timeout 240`, so a stuck install fails fast and auto-recovers instead of burning the runner to the job timeout — the job still fails if all attempts are exhausted. See the CI-hardening note above.
- **The kind + KWOK harness lives at `internal/testcluster`** (`testcluster.New(t, ...)`, from issue #5). It programmatically spins up a real single-node kind cluster and attaches a KWOK controller — **Docker is the only prerequisite** (no `kind`/`kwok` CLI, no pre-existing cluster). It ran live in ~45s in the build environment. Teardown is registered via `t.Cleanup` before creation, so clusters are always removed even on panic/failure. Use `AddFakeNodes`/`AddFakePods` for scale without real containers.
- **KWOK manifests are vendored** from pinned upstream v0.8.0 under `internal/testcluster/manifests/` (with attribution). Don't hand-edit them casually; re-vendor from a pinned upstream tag if they need updating.
- **The fake clientset (`k8s.io/client-go/kubernetes/fake`) drives SharedInformers fine** — creating/updating/deleting objects through it delivers watch events to informers. That's how `SceneWatcher`'s "watch events → Scene Deltas" unit tests (`internal/kube/watch_test.go`) run without a real cluster. Use a timeout-bounded channel read for the async informer→worker→broadcast hop (a few seconds is plenty).
- **For deterministic real-cluster pod-lifecycle deltas, bind the pod to a *bare* Node (no KWOK annotation, no kubelet) and drive its status yourself.** The `SceneWatcher` integration test (`watch_integration_test.go`) needs a *separable* create → update → delete so each yields its own delta; a KWOK-managed node flips Pending→Running so fast it **coalesces the create+update into a single `panelAdded`** (rebuild-and-diff reports net change), so there's no distinct `panelUpdated`. On a bare node nothing races you: create → `panelAdded`, then `UpdateStatus` phase=Running → `panelUpdated`. **Two gotchas of bare (kubelet-less) nodes:** a bound pod stays `Pending` until you set its status (fine — the panel appears regardless of phase), and **`Delete` hangs in `Terminating` forever** (no kubelet completes graceful deletion) — force-delete with `GracePeriodSeconds: 0` so `panelRemoved` fires. Assert by reconstructing the scene (snapshot + applied deltas) and waiting for convergence, not on exact per-event deltas (a real watch coalesces/races).

## Parallel-dispatch hazards

- **Two tickets that both edit `go.mod`/`go.sum` will conflict when the second merges**, even with no source overlap. This happened with PRs #35 (added gorilla/websocket) and #36 (added kind/KWOK deps). Resolution: merge `main` into the lagging branch, take one side's `go.mod`/`go.sum` wholesale, re-run `go mod tidy`, then **re-verify `govulncheck` and the x/net floor survived**. Prefer scoping parallel tickets so only one touches Go dependencies at a time.
- **File-ownership boundaries prevent most conflicts.** When dispatching parallel work, give each agent an explicit disjoint file scope (backend → `cmd/`+`internal/`, frontend → `web/`, CI → `.github/`, etc.) and name the sibling ticket's scope so it stays clear.
- **`.claude/agents/*.md` custom subagent types may not be available in every runtime.** When they aren't, dispatch a `general-purpose` agent and fold the role's persona (from the corresponding `.claude/agents/*.md`) into the prompt instead.
- **Close issues on PR *merge*, not PR *open*.** Native issue dependencies count *open* blockers, so closing an issue when its PR opens makes downstream tickets read as unblocked before their dependency's code is on `main`. Use `Closes #<n>` in the PR body so GitHub auto-closes on merge; the orchestrator gates the frontier on "blocker PR merged," not "issue closed." See `docs/agents/issue-tracker.md`.
- **The e2e (Playwright) job must provision a real cluster.** Since issue #9 the binary *fails startup* when it cannot reach a Kubernetes cluster. Playwright's `webServer` launches that binary, so the e2e job now creates a single-node `kind` cluster (`helm/kind-action`, pinned to SHA) and exports `KUBECONFIG` to `~/.kube/config`; `web/playwright.config.ts`'s `webServer.env` passes `KUBECONFIG` through so the launched binary connects (client-go honors `KUBECONFIG` / `~/.kube/config`). A fresh kind cluster makes the CI user cluster-admin, so the app detects Node view mode. Do NOT weaken the fail-on-unreachable startup to dodge this — provisioning a cluster is the intended fix (ADR-0004). The smoke test asserts a *well-formed* `/ws` frame (parsed JSON, not a frozen string) so it survives the wire format growing: as of issue #10 that frame is a `SceneState` snapshot `{viewMode:"node"|"namespace"}` (no `type` field — the ad-hoc `{type:"viewMode",...}` message was replaced), and the test resolves on the first frame with a valid `viewMode` without over-constraining the rest.
- **Two individually-green PRs can conflict *semantically* once both land on `main`** — not just via file/`go.mod` overlap. The e2e harness (issue #7) launched `./bin/htp-k8s` back when it was a clusterless placeholder; issue #9 later made the binary fail startup without a reachable cluster. Each PR was green alone, but together they broke the e2e check on every PR (a docs-only PR, #46, first surfaced it). Lesson: when app behavior changes in a way that affects *how it is launched* (startup preconditions, required env, ports, config), check and update the e2e harness (`web/playwright.config.ts`, `.github/workflows/pr.yml` e2e job) in the same change.
